## Problem Statement
用户希望系统掌握 macOS 全键盘操作,但当前缺少同时覆盖 Apple 全家桶 + 常用三方 app 的结构化练习工具——系统自带的"快捷键速查"是静态文档,第三方游戏化卡牌只覆盖系统,都不能满足"真去 Mac 里练 + 双向抽考"的需求。

## What Makes This Cool
一本"全键盘百科全书",按 Apple User Guide 的章节结构分板块,把 macOS 系统、Finder、Terminal、文本编辑、VSCode、Chrome 的快捷键整理成可勾选的清单 + 双向随机抽考。用户去真 Mac 里练、回来勾,app 不画任何假 UI,只做引导和复习——工具该有的样子。

## Constraints
- 单 Mac 个人工具,无账号、无云同步、无分享、不上架 App Store
- 第一次覆盖范围:系统 + Finder + Terminal + 文本编辑 + VSCode + Chrome,固定 6 个板块(v1 Sidebar 无"+"按钮,自定义留到 v2)
- 内容筛选标准:每个 app 内 P0(每天用)全收、P1(每周用)选收,v1 总数控制在 180–200
- 快捷键数据固化在 app bundle 的 JSON 里,改内容需要出新版本 app
- 进度 + 复习历史 + 错题计数 = 全部存当前 Mac 的 UserDefaults,不跨设备
- 复习 = 从 `practiced ∪ recent_mistakes` 集合随机抽,50% 概率"看键选功能(4 选 1)"、50% 概率"看功能手输键"
- Swift + SwiftUI 原生 macOS app,Deployment Target = macOS 14.0
- 全部 UI 用 SwiftUI semantic colors(`Color.primary/.secondary/.accentColor`)和 system materials,不写硬编码颜色
- 三栏布局:Sidebar(板块)+ Middle(快捷键列表)+ Detail(详情 + 勾选框)

## Premises
1. 单 Mac 个人工具,无需账号、云同步、分享功能。
2. 首次覆盖范围:系统 + Finder + Terminal + 文本编辑 + VSCode + Chrome,固定 6 板块。
3. 快捷键数据手工从 Apple User Guide + VSCode/Chrome 官方键位表整理,固化在 bundle JSON 里(启动时加载)。
4. 进度 + 复习历史 + 错题计数全部存当前 Mac UserDefaults,无云端。
5. 复习循环:从 `practiced ∪ recent_mistakes` 集合随机抽,50% 看键选功能(4 选 1)、50% 看功能手输键。
6. UI 用 SwiftUI NavigationSplitView 做三栏,⌘F 搜索,暗色模式自动适配,语义色 + system materials。
7. 完整 v1 工程量约 5 周(单人在 Mac 上,含 1 周 buffer;数据录入作为独立轨道与编码并行)。

## Data Schema
快捷键 JSON 顶层结构:
```json
{
  "version": 1,
  "lastUpdated": "2026-06-24",
  "shortcuts": [
    {
      "id": "sys.spotlight.open",
      "category": "system",
      "subcategory": "spotlight",
      "displayKey": "⌘Space",
      "description": "打开 Spotlight 搜索",
      "appScope": ["system"],
      "verifiedAgainst": "macOS 14 Apple User Guide",
      "verifiedDate": "2026-06-20",
      "aliases": ["Cmd+Space"]
    }
  ]
}
```

字段规则:
- `id`:小写 + 点分隔,人类可读,稳定不变
- `displayKey`:必须用 Unicode 符号 `⌘ ⇧ ⌥ ⌃ ⌫ ⏎ ⇥`,**禁止** `Cmd/Ctrl/Alt` 英文写法
- `appScope`:数组,值域 `["system", "finder", "terminal", "text", "vscode", "chrome"]`
- `aliases`:同义英文写法数组,用于搜索 + 手输判分容错
- 仓库内附 `docs/shortcuts.schema.json`(JSON Schema draft 2020-12 形式化),加载时用 `JSONDecoder` 配合 schema 校验,失败则拒启动

Normalization 规则(加载时一次,运行时复用):
- `Cmd` → `⌘`、`Shift` → `⇧`、`Ctrl` → `⌃`、`Option/Alt` → `⌥`、去空格和 `+`
- 归一后做精确匹配;若 `aliases` 归一结果与多条 displayKey 冲突,加载时打印 warning 并拒启动
- displayKey 中多键位固定按 `⌃ ⌥ ⇧ ⌘` 顺序排(如 `⌘⇧[` 而非 `⇧⌘[`)

## Data Sourcing & Verification
- 核验人:用户本人
- 来源清单(写入 `docs/sources.md`):
  - Apple User Guide(macOS 14.x): `https://support.apple.com/guide/macbook/welcome/mac`
  - VSCode Keybindings: `https://code.visualstudio.com/docs/getstarted/keybindings`
  - Chrome Keyboard Shortcuts: `https://support.google.com/chrome/answer/157179`
- 核验节奏:每年 macOS 大版本发布后 2 周内做一次全量复核
- 复核任务:`JSON 字段 verifiedDate > 12 个月`的条目须人工再核验

## Review Logic
每条快捷键有 4 种状态,严格状态机:
```
unseen ──[勾选已练]──> practiced
unseen ──[任意错误]──> practiced          (防御性,正常流程不会触发)
practiced ──[答对 1 次]──> mastered
practiced ──[该 id 错误次数累计 ≥ 2]──> recent_mistakes
mastered ──[答错]──> practiced
recent_mistakes ──[答对 1 次]──> practiced
```
- 抽考候选集合:`practiced ∪ recent_mistakes`(不抽 `mastered` 也不抽 `unseen`)
- 50% 概率"看 displayKey 选 description(4 选 1)";50% 概率"看 description 手输 displayKey"

4 选 1 干扰项生成规则:
1. 优先从同 `category + subcategory` 全集抽 3 个 description(不限于已练过的)
2. 同 `category + subcategory` 不足 3 个 → 补到同 `category` 全集
3. 同 `category` 仍不足 3 个 → 补到全库
4. 选项顺序每次随机 shuffle
5. 排除正确答案,任何 3 个不能重复

手输键判分算法:
1. 标准化输入:大小写不敏感、去空格、修饰键归一(`Cmd→⌘` 等,同加载 normalization)
2. 输入归一后与每条 `displayKey` 比对
3. **同时**与 `aliases` 归一结果比对(用户输入 `Cmd+Space` 等价于 `⌘Space`,判对)
4. 键位顺序必须完全一致(不做 typo 容忍,不做同义键容忍)
5. 判错显示正确答案,该 id 错误计数 +1

错误计数存 UserDefaults key `mistakeCount[id]`,加载时一并读入内存。

## UI States
三栏布局(最小窗口 900×600):
- Sidebar 宽 180–240pt:固定 6 板块(系统 / Finder / Terminal / 文本编辑 / VSCode / Chrome),板块名后显示"(已练 12/30)"——12 = `practiced + mastered + recent_mistakes` 三状态总数
- Middle 宽 280–360pt:当前板块的快捷键列表,每行 = displayKey + description 缩略 + 勾选状态
- Detail 宽 ≥300pt:
  - 顶部:大字号 displayKey(60pt SF Mono)
  - 中部:description 完整描述 + 适用 app + 核验来源
  - 中下部:"我练过了"勾选框、"显示/隐藏 aliases"切换
  - 底部:本次状态(unseen / practiced / mastered / recent_mistakes)+ 错误次数
- 顶 toolbar:搜索框(⌘F 聚焦)+ 复习按钮(⌘R 触发)
- Sidebar 顶端有"全部 / 已练 / 错题"快速过滤段
- v1 Sidebar **不预留** "+" 按钮,自定义板块留 v2

搜索行为:单一输入框,跨字段模糊匹配 `displayKey + description + aliases + category`,实时过滤 Middle 列表,无需按回车。

## Approaches Considered

### Approach A: 极简 MVP
- Summary: 单 Xcode 项目、1 个 JSON(~50 个最常用快捷键)、3 个 Swift 文件、基础三栏 SwiftUI、只做"看键选功能"一种考题。
- Effort: S(1 个周末)
- Risk: Low
- Pros: 1 周末可交付、JSON 改一行就能迭代、出问题只影响 50 条数据
- Cons: 内容只覆盖 50 个最常用,达不到 v1 全范围;考题只有单向,不能练手输键;缺搜索、缺复习历史
- Reuses: 无(从零开始)

### Approach B: 完整 v1(已选,渐进式 5 节点交付)
- Summary: MVVM + `@Observable` 多文件项目,完整 D5 范围(系统+Finder+Terminal+文本编辑+VSCode+Chrome,渐进式 50→100→150→180→200)、抽考分两阶段(第 4 周先单向 4 选 1、第 5 周加双向随机)、⌘F 搜索、暗色模式适配。
- Effort: L(4 周开发 + 1 周 buffer = 5 周 v1.0 硬截止)
- Risk: Med
- Pros: 一次到位交付 D1–D9 全部要求;后续 v2 加 Safari/Mail/Notes 时只加 JSON 不改代码;双向考题覆盖更全
- Cons: 工程量比 A 大几倍;MVVM 拆分要规划清楚;内容 200 条要时间从官方文档整理(数据录入与编码并行)
- Reuses: SwiftUI NavigationSplitView(原生三栏)、Foundation JSONDecoder(系统)

### Approach C: 内容优先 + 真实 app 跳转
- Summary: 不在 app 里画考题,只在三栏里展示快捷键说明 + 一个"现在就练"按钮,点击后 NSWorkspace 打开对应 app(只限 Finder 和 Terminal,VSCode/Chrome 不在 v1 自动跳转范围),用户去真 app 里练,回来勾"已练"。只在右侧详情面板用 ⌘R 抽 1 题纯手输——保留最弱的复习形式,核心是"自评 + 查阅"。
- Effort: M(1–2 周)
- Risk: Med-High
- Pros: 最贴合"我直接去 Mac 里练";UI 最简
- Cons: 没有主动抽考循环,复习靠自评不强制;VSCode/Chrome 安装路径不确定,需要写 fallback
- Reuses: NSWorkspace(打开系统 app)

## Recommended Approach
Approach B(完整 v1)。理由:D1–D9 已经把功能定得很全,只有 B 能完整交付;A 太小,1 个月内会不够用;C 砍掉了"主动抽考循环",只剩下自评和查阅。完整 v1 的工程量(5 周含 buffer)对于一个会每天用的个人工具来说,合理且必要。数据手工录入作为独立轨道,可与编码并行,缩短关键路径。

## Risk & Buffer
Top 3 风险与 mitigation:
1. **数据录入超期**:200 条手工录入 + 核验可能超 1 周。Mitigation:第 1 周就开始并行录入轨道,数据先入草稿 JSON,核验延后;条目数降级 180 也接受
2. **双向抽考 UX 返工**:手输键判分逻辑用户体感差可能返工。Mitigation:第 4 周末提前在真机上 dogfood 1 次,根据体验决定是否在第 5 周继续
3. **macOS 14 NavigationSplitView 兼容性**:少数布局细节在 14.0/14.5 间有差异。Mitigation:第 2 周搭完 UI 后立即在 14.0 跑一遍,留 1 周 buffer 调

## Open Questions

- `displayKey` 中包含"功能键"如 F1–F12 是写 `F1` 还是 `fn+F1`(MBP 触控栏机型)?Default: 写 `F1`,Decide by: 第 1 周录入时遇到再说

- 错题本是否提供"标记为已掌握"手动摘出按钮?Default: v1 不提供(累计错≥2 进 recent_mistakes,答对 1 次自动回到 practiced),v2 再加,Decide by: v1.0 dogfood 后

## Success Criteria
- v1.0 发布时快捷键总数 ≥180 条
- 打开 app 三栏布局正常,⌘F 搜索跨字段模糊匹配,⌘R 触发抽考
- 数据从 bundle JSON 加载,启动时做 normalization 和 schema 校验
- 勾选"我练过了"后状态持久化,重启后保留
- 复习出题符合 Review Logic 中的状态机 + 4 选 1 干扰项生成规则
- 手输键判分按定义算法,aliases 归一也算对
- 暗色模式自动适配,语义色 + system materials
- 单元测试覆盖 JSON 加载 + normalization、状态机转移、手输键判分、4 选 1 干扰项生成
- 目标 line coverage ≥70% for `Model/` 和 `Review/`
- "1 人 1 个月内学完 v1" 的可观测定义:勾选全部 180+ 条 + 30 天内对全部 `mastered` 状态条目做 3 轮双向抽考,平均错题率 < 15%

## Test Strategy
v1 必须有单元测试覆盖以下 4 块(其余不要求):
1. **JSON 加载 + normalization + schema 校验**:加载 bundle JSON 后,每条 displayKey 都能被其 aliases 反向归一命中;schema 校验失败的 JSON 拒启动
2. **状态机转移**:所有 4 种状态之间的合法转移 + 防御性 `unseen → practiced`;非法转移必须拒绝
3. **手输键判分**:大小写不敏感、修饰键符号归一、aliases 归一、键位不一致判错、冲突检测
4. **4 选 1 干扰项生成**:同 `category + subcategory` 优先、3 级 fallback、不重复、不命中正确项

不测 UI、不测 performance、不测兼容性。MacTrainerTests target 用 Swift Testing(`import Testing`)写。目标 line coverage ≥70% for `Model/` 和 `Review/`。CI(本地 Xcode Test)失败则 v1.0 不发布。

## Distribution Plan
本地构建,不在 App Store 上架,不上签名,不走分发渠道。Deployment Target = macOS 14.0。流程:
1. 在 Mac 上 git clone 项目,用 Xcode 打开 `MacTrainer.xcodeproj`
2. Product > Archive > Export > macOS App
3. 把生成的 `.app` 拖到 `/Applications/`

未来如果想分发给其他人,再加 GitHub Releases + Sparkle 自动更新。本 v1 不做。

## What I noticed about how you think
- 你 D1 直接说"不需要你模拟使用场景",这就排除了我准备给你的最复杂方案(造一个假 Finder)。这说明你分得清"app 干什么"和"你干什么"——不要重复造你已经能做的轮子
- 每一题你都选了"推荐"或最简路线,显示出对工程量边界的判断力。能分辨"今天就要"和"以后再说"——这种取舍比技术选型本身更稀缺
- D5 你自定义答"系统+finder+终端+文本编辑+vscode+chrome",没用预设选项,说明你清楚自己最常用什么,不要为了"完整"硬塞用不上的 app
- "我直接在 mac 中启动需要的场景练习"——你给 app 划清了职责边界:它是引导和复习工具,不是仿真器。这种边界感是好的工程直觉
